## 背景と現状

第一次リファクタリングで `src/client/main.ts` は約 3000 行まで減ったが、その後の新機能
(ポーズ編集・CIGPose・複数人ポーズ・ペイントツール・LLM プロンプト改善・エッジポップアウト等)が
すべて main.ts に追加され、**6012 行**まで再膨張した。`styles.css` も **3791 行**ある。

### 再膨張の構造的な原因

1. 第一次は「pure helper の抽出」が中心で、**state に触る orchestration は main.ts に残す**方針だった。
2. その結果 `state` / `render()` / イベント配線に触れられるのが main.ts だけになり、
   **新機能の stateful な部分は main.ts に書くしかない構造**になっている。
3. `bindEvents` / `handleAction` が全機能の中央ハブで、機能追加のたびに必ず肥大化する。
4. ファイルサイズを機械的に見張るガードがない。

つまり「また減らす」だけでは同じことが再発する。今回は **state と配線の構造を変える**。

## 方針(アグレッシブ許容)

1. **state の分離と依存の逆転**: `state` オブジェクト・型を `src/client/appState.ts` に移し、
   各 controller モジュールが `import { state } from "./appState"` する。
   再描画は `requestRender()`(callback 登録式)を appState 側に置き、循環 import を断つ。
2. **ドメインごとの controller 分割**: 下表のモジュールへ移動。関数本文は verbatim 移動を基本とし、
   挙動変更を混ぜない。
3. **action registry 化**: `handleAction` の巨大 switch を `Map<string, (id, target) => void>` にし、
   各 controller が自分の action を `registerActions({...})` で登録する。
   `bindEvents` も各 controller の `bindXxxEvents(app)` に分割する。
   → **以後の新機能は main.ts に 1 行も足さずに追加できる**ことがゴール。

## CSS 計画(Chrome 150 前提、フェーズ J)

`styles.css`(3791 行)は prefix こそほぼ無いが、Chrome 150 で使える機能で大きく整理できる:

1. **CSS Nesting**: 長い子孫セレクタの繰り返し(`.mask-toolbar …` 系など)をネストにまとめる。
   行数と selector 重複が大きく減る。
2. **`color-mix()` によるトークン派生**: `rgba(255,255,255,…)` が 51 箇所、
   accent 系 `rgba(226,173,129,…)` の生値が複数ある。
   `color-mix(in srgb, var(--ink) 9%, transparent)` / `color-mix(in srgb, var(--accent) 42%, transparent)`
   の形に置換し、`--accent-soft/-line/-fill` も `:root` で color-mix 定義にする。
   → accent 変更が 1 箇所で全体へ波及する。
3. **CSS Anchor Positioning**: `.iteration-edge-popout` の位置計算
   (`main.ts` の `positionIterationEdgePopout` + mouseover ハンドラ、fixed 配置ハック)を
   `anchor-name` / `position-anchor` / `position-area` / `position-try-fallbacks` に置換し、**JS を削除**。
   「下→上フォールバック・端クランプ」は position-try がネイティブでやってくれる。
4. **Popover API**: action ドロップダウン(`closeOpenActionDropdowns` による手動 light-dismiss)を
   `popover="auto"` + anchor positioning に置換し、JS の外側クリック処理を削除。
5. **`@scope`**: モーダル / パネル単位のスコープ化で衝突回避の長いセレクタを短縮。
6. **`interpolate-size: allow-keywords`**: サイドバー折りたたみ等の height:auto 遷移を CSS だけで。
7. **ファイル分割**: `tokens.css` / `base.css` / `components/*.css` に分け、esbuild(`scripts/build.mjs`)で
   bundle するか `@import`(ローカル配信なので許容)。
8. **フォントの自己ホスト**: Google Fonts の `@import` はオフライン時にフォールバックフォントになる。
   ローカルアプリなので woff2 を `public/` に置いて `@font-face` 化する(UX 改善 #8 と同件)。

CSS 変更は視覚回帰が起きやすいので、主要画面(Home / Project / 画像詳細 / マスク編集 / ポーズ編集)の
before/after スクリーンショット比較をフェーズ内で行う。

## UX 改善候補(別フェーズ、優先度順)

1. **`render()` の全画面 innerHTML 再構築をやめる**(大工事・最優先候補)。
   現状は毎回 `app.innerHTML` を組み直すため、スクロール保存/復元・rAF 二重復元・
   クリック選択の遅延タイマーなど対症療法が積み重なっている。領域別レンダラ
   (ヘッダー / ツリー / グリッド / モーダル)への分割 + 差分更新にすれば、
   フォーカス消失・画像ちらつき・スクロールジャンプが根本から消える。
   ※ main.ts 分割(A〜I)完了後の独立プロジェクトとして実施。
2. **シングルクリックの人工遅延の解消**: asset card / iteration dot は dblclick と区別するため
   タイマーで選択を遅延しており、クリック反応が鈍い。ダブルクリック機能を別 UI
   (ボタン / 修飾キー)に移して即時選択にする。
3. **Round 削除(`deleteRoundTree`)に確認がない**: Project / Template 削除は confirm があるのに、
   ツリーごと消す Round 削除はダブルクリック→プレビュー経由のみ。confirm か Undo(遅延実行 +
   トーストで取り消し)を付ける。
4. **トーストの改善**: `state.message` は単一文字列なので連続操作で上書きされ、
   info と error の区別もない。種類別スタイル + スタック表示、エラーは自動クリアしない。
5. **生成進捗の可視化**: サーバは ComfyUI WebSocket で進捗を見ているのに、UI は 3 秒 polling の
   枚数増加だけ。round dot / 生成ボタンに進捗 %(または現在 step)を出す。
6. **キーボードショートカットのヘルプ**: `?` で一覧オーバーレイ。マスク / ポーズ編集の
   ショートカットが増えており発見性が低い。
7. **エッジポップアウトの hover 依存**: タッチ・キーボードで出せない。anchor positioning 化
   (CSS #3)と同時に focus / クリックでも表示できるようにする。
8. **フォント自己ホスト**(CSS #8 と同件): オフラインでも見た目が変わらないように。

## テスト手法

### 自動テスト

- `npm run typecheck`(strict, 0 エラー維持)
- `npm test`(現在 281+ 件。リファクタリングでは件数を減らさない)
- `npm run build` / `npm run check`
- テスト DB 必須: `GURUGURU_TEST_DB=1`、データは外部 `GURUGURU_TEST_DATA_DIR`

### テスト用 ComfyUI(ダウンロード済み Desktop 版の流用)

生成パイプライン(workflow patch・ControlNet・unified switch)の実機検証は、
**本番 ComfyUI(8188)には触らず**、DL 済み Desktop 版 ComfyUI のコードと Python 環境を流用した
分離インスタンス(port 8288)で行う。実績: 2026-07-03 の UnifiedSwitch ControlNet 検証。

- コード: `$env:LOCALAPPDATA\Comfy-Desktop\ComfyUI-Installs\ComfyUI\ComfyUI`(v0.20.1)
- Python: 同ディレクトリの `standalone-env\python.exe`(Python 3.13 / torch 2.10 cu130)
- 共有モデル定義: `%APPDATA%\Comfy Desktop\shared_model_paths.yaml`

手順:

```powershell
# 1. リポジトリ外にテスト用 base ディレクトリを用意(custom_nodes が無いと起動時 FileNotFoundError)
$base = "$env:TEMP\comfy-test-8288"
New-Item -ItemType Directory -Force "$base\custom_nodes", "$base\input", "$base\output", "$base\temp"
# unified switch 検証では未使用枝の LoadImage にも実在ファイルが必要 → input にダミー画像を置く

# 2. 起動(モデルは Desktop 版と共有)
$comfy = "$env:LOCALAPPDATA\Comfy-Desktop\ComfyUI-Installs\ComfyUI"
& "$comfy\standalone-env\python.exe" "$comfy\ComfyUI\main.py" `
  --port 8288 --base-directory $base `
  --extra-model-paths-config "$env:APPDATA\Comfy Desktop\shared_model_paths.yaml"

# 3. 疎通確認(必ず --max-time 付き)
curl --max-time 5 http://127.0.0.1:8288/system_stats
```

GURUGURU 側はテストインスタンス(`PORT` ≠ 5177・`GURUGURU_TEST_DB=1`・外部
`GURUGURU_TEST_DATA_DIR`)を起動し、設定画面の ComfyUI URL に `http://127.0.0.1:8288` を入れて接続する。
GPU 負荷を抑えるため検証は低負荷パラメータ(例: 512x288 / 6 steps)で、
txt2img → img2img → inpaint → pose 添付の順に 1 枚ずつ生成して確認する。
