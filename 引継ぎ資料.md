## 最重要の安全ルール

- 本番ユーザーデータを調査・要約・読み込みしない。
- `app.db`、生成画像、プロンプト、ワークフロースナップショット、サムネイルなどの runtime data はリポジトリ外に置く。
- DB を初期化しうるコマンドやサーバー起動は、検証時に必ず `GURUGURU_TEST_DB=1` を付ける。
- 安定したテスト用保存先が必要な場合は、リポジトリ外の `GURUGURU_TEST_DATA_DIR` を指定する。
- Codex がアプリを起動する場合、既定ポート `5177` は使わず、明示的に別 `PORT` を指定する。
- `git reset --hard`、`git checkout --`、未確認の破壊的削除は使わない。
- 既存の未コミット差分があれば、ユーザー変更として扱い、勝手に戻さない。

## 状態管理と UI

- client は framework なしの単一 SPA。
- `src/client/main.ts` の `state` object が主要な UI 状態を保持する。
- DOM は `render()` で `app.innerHTML` をまとめて再生成する。
- イベントは `bindEvents()` で document/app level に集約されている。
- iteration tree は DOM 再生成時に scroll position を保存・復元する設計。
- mask editor/WebSAM は同じ `main.ts` 内で、canvas layer、worker message、draft state を直接管理している。

## API/生成/ファイル処理の責務

- `routeApi()` が `/api/*` を一括 dispatch している。
- project handler は `server/projects.ts`、template handler は `server/templates.ts`、asset handler は `server/assets.ts`、round/generation handler は `server/rounds.ts`、source asset upload handler は `server/sourceAssets.ts` に分離済み。`server/index.ts` は route dispatch と settings/WebSAM/shutdown のみ。
- generation は `generation_rounds` と `generation_jobs` に分かれる。
- 通常生成は GURUGURU 側で `batch_size=1` の job に分割し、ComfyUI に複数 prompt を queue する。
- server は ComfyUI WebSocket と UI 側 collect polling の両方で生成結果を回収する。
- asset/mask の実ファイルは `dataRoot/projects/<projectId>/...` に保存される。
- static file serving は `server/files.ts`、asset file serving は `server/assets.ts` から `server/files.ts` / `server/storage.ts` を使う構成。
