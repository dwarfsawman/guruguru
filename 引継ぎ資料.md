## 最重要の安全ルール

- 本番ユーザーデータを調査・要約・読み込みしない。
- `app.db`、生成画像、プロンプト、ワークフロースナップショット、サムネイルなどの runtime data はリポジトリ外に置く。
- DB を初期化しうるコマンドやサーバー起動は、検証時に必ず `GURUGURU_TEST_DB=1` を付ける。
- 安定したテスト用保存先が必要な場合は、リポジトリ外の `GURUGURU_TEST_DATA_DIR` を指定する。
- Codex がアプリを起動する場合、既定ポート `5177` は使わず、明示的に別 `PORT` を指定する。
- `git reset --hard`、`git checkout --`、未確認の破壊的削除は使わない。
- 既存の未コミット差分があれば、ユーザー変更として扱い、勝手に戻さない。

## リポジトリ構成

- `src/server/index.ts`: Node HTTP サーバーのエントリポイント。API routing、project/template/round/asset 操作、generation queue、ComfyUI collection、static/file serving が集中している。
- `src/server/db.ts`: SQLite 初期化、設定、schema migration、row 変換、data directory 解決。
- `src/server/comfy.ts`: ComfyUI HTTP/WebSocket 連携、prompt queue、history/view/upload/interrupt。
- `src/server/storage.ts`: project storage、画像/マスク保存、asset file stream、project storage 削除。
- `src/server/paths.ts`: ファイルパス境界判定（`isPathInside` / `isPathInsideOrEqual`）。
- `src/server/workflow.ts`: ComfyUI workflow patch、seed 解決、role map 正規化、inpaint/img2img workflow 変換。
- `src/client/main.ts`: SPA の状態、イベント、API 呼び出し、render、mask editor、WebSAM 制御の大半。
- `src/client/icons.ts`: 第1フェーズで抽出した SVG icon helper。
- `src/client/websam/*`: WebSAM / SlimSAM-77 の model 定義、worker 型、ONNX Runtime worker。
- `src/shared/*`: client/server 共有型、constants、workflow diagram、role map 推論/検証。
- `src/shared/generationMode.ts`: generation mode ごとの denoise、parent asset 要否、parent relation 変換。
- `scripts/build.mjs`: esbuild による server/client/worker bundle と static asset copy。

## エントリポイント

- server: `src/server/index.ts`
- client: `src/client/main.ts`
- WebSAM worker: `src/client/websam/worker.ts`
- build script: `scripts/build.mjs`
- public HTML: `src/client/index.html`

## 状態管理と UI

- client は framework なしの単一 SPA。
- `src/client/main.ts` の `state` object が主要な UI 状態を保持する。
- DOM は `render()` で `app.innerHTML` をまとめて再生成する。
- イベントは `bindEvents()` で document/app level に集約されている。
- iteration tree は DOM 再生成時に scroll position を保存・復元する設計。
- mask editor/WebSAM は同じ `main.ts` 内で、canvas layer、worker message、draft state を直接管理している。

## API/生成/ファイル処理の責務

- `routeApi()` が `/api/*` を一括 dispatch している。
- template/project/round/asset の DB 操作は `server/index.ts` 内の関数に分散している。
- generation は `generation_rounds` と `generation_jobs` に分かれる。
- 通常生成は GURUGURU 側で `batch_size=1` の job に分割し、ComfyUI に複数 prompt を queue する。
- server は ComfyUI WebSocket と UI 側 collect polling の両方で生成結果を回収する。
- asset/mask の実ファイルは `dataRoot/projects/<projectId>/...` に保存される。
- static file と asset file serving は `server/index.ts` と `server/storage.ts` にまたがる。

- DB row/API row は `Record<string, unknown>` と `toApiRow()` に大きく依存している。
- `JSON.parse(... as T)` や手動 cast が複数ある。
- runtime validation は局所的で、request/response 全体を schema として固定していない。
- client と server で似たルールが別実装になっている。

重複が確認された helper:

- `defaultDenoiseForMode`: client/server → P1 で `src/shared/generationMode.ts` に集約済み

- `nodeIdFromRolePath`: `server/index.ts` / `server/workflow.ts`

## 次フェーズ候補

### P0: path containment の安全性整理 (完了)

対象:

- `src/server/storage.ts`
- `src/server/index.ts`
- `src/server/db.ts`

実施内容:

- 新規: `src/server/paths.ts` に `isPathInside`（厳密: target が parent の真の子）と `isPathInsideOrEqual`（同値も含む）を集約。どちらも `relative(resolve(parent), resolve(target))` ベース。
- `storage.ts` の局所 `isPathInside` を削除し import に切替。`safeFileStream` の大小文字無視 `startsWith` prefix 判定を `isPathInside` へ寄せた。
- `index.ts` の `serveStatic`（publicDir 境界）と `streamFile`（dataRoot 境界）の `startsWith` prefix 判定を `isPathInside` へ寄せた。
- `db.ts` の起動時 data dir 拒否は同値ケース（data dir == repo root）も拒否したいので `isPathInsideOrEqual` を使用。局所 `isPathInside` は削除。
- 挙動差: `safeFileStream`/`serveStatic` で「target == parent 丁度」は以前は通過→stream error だったが、厳密判定で境界外→fallback/throw になる。実運用（ディレクトリそのものを配信しない）には影響しない。

検証済み:

- `GURUGURU_TEST_DB=1 npm run build`: 成功
- `GURUGURU_TEST_DB=1 npm run check`: 成功
- `git diff --check`: 成功

## 作業時の推奨手順

1. `git status --short` でユーザー差分を確認する。
2. 対象領域を `rg` と小さな file read で調べる。
3. 挙動変更がない抽出か、挙動変更を含む修正かを分ける。
4. 本番 data dir を読まないよう、DB 初期化が絡むコマンドは `GURUGURU_TEST_DB=1` を付ける。
5. 変更後は `git diff --check` と該当する npm script を実行する。
6. coherent な編集単位で commit する。
