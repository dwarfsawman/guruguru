# 操作メモ

GURUGURU の操作方法や注意点を追記するためのメモです。大きな仕様書ではなく、実際に開発・運用するときに必要な手順と落とし穴を残します。

## 開発時の基本操作

- 依存関係は `npm install` で用意します。
- ビルドは `npm run build` を使います。
- 開発起動は `npm run dev` を使います。これはビルド後に `dist/server/index.js` を起動します。
- 構文チェックを含む確認は `npm run check` を使います。
- 実行には Node.js 24 以上が必要です。

## データ保存場所

- 通常実行時のユーザーデータは OS のユーザーデータディレクトリに保存されます。Windows では `%LOCALAPPDATA%\GURUGURU` が既定です。
- `GURUGURU_DATA_DIR` を指定する場合は、必ずこのリポジトリ外のパスにします。
- `app.db`、生成画像、プロンプト、ワークフロースナップショット、サムネイルなどのランタイムデータはリポジトリに置きません。
- リポジトリ直下や `data/` に実運用データベースを作らないでください。

## テスト・検証時の注意

- データベースを初期化するコマンドやテストでは `GURUGURU_TEST_DB=1` を設定します。
- テスト用データの保存先を明示したい場合は `GURUGURU_TEST_DATA_DIR` を使います。
- 本番のユーザーデータディレクトリや実運用の `app.db` を、テスト・調査・要約目的で読み込まないでください。
- コマンドがリポジトリ内のデータディレクトリを拒否した場合は、外部のテスト用ディレクトリを指定してやり直します。
- 既定ポート `5177` はユーザーが `npm start` などで起動する通常インスタンス用に空けておきます。Codex が検証用にアプリを起動する場合は `PORT` に 5177 以外を指定し、`GURUGURU_TEST_DB=1` とリポジトリ外の `GURUGURU_TEST_DATA_DIR` を使います。
- 複数インスタンスを同時に動かす場合は、ポートだけでなくデータディレクトリも分けます。同じ本番データディレクトリを複数プロセスから使う運用は避けます。

## 画像マスク・inpaint

- 画像詳細ビューアのマスク編集は、親画像と同じサイズの PNG マスクだけを生成リクエストに使います。
- GURUGURU が生成するマスク PNG は透明背景に白で描画します。ComfyUI の `LoadImageMask` は alpha を反転マスクとして読むため、workflow patch では `red` チャンネルを使い、白い描画部分を更新対象にします。
- `generation_rounds.request_json` には `maskedContent`、`inpaintArea`、`onlyMaskedPadding` と保存済みマスク参照だけを残し、巨大な `maskDataUrl` は保存しません。
- マスク画像は各 Project の外部データディレクトリ配下 `projects/<projectId>/masks/` に保存されます。本番DBや本番データを調査せず、検証時は必ず外部テストデータディレクトリを使ってください。

## プレビューダイアログの最小化・移動

- 画像詳細ビューアの `.mask-toolbar`（プロンプト・マスク設定）は邪魔にならないよう最小化・移動できる。
- `state.maskToolbarMinimized` が true のとき本体を隠し、ヘッダーと展開ボタンだけを残す。
- ダイアログ表示中（非最小化）は `.mask-canvas` に `mask-locked` を付け `pointer-events: none` にし、マスク描画を無効化する。最小化するとマスクが効く。
- マスク編集モードに入った直後はツールバーを自動で最小化する。縦長画像では展開済みツールバーが画像全体を覆い、プロンプト等の文字要素が選択されてしまうため。プロンプト・設定を編集したいときは展開ボタンで戻す。
- ダイアログ外（画像・背景など）のタップで最小化する。ただし `[data-action]` や `.mask-toolbar` 内のタップは最小化しない。最小化済みのとき背景タップは従来通りモーダルを閉じる。
- ヘッダー (`[data-mask-toolbar-handle]`) のドラッグで `position: fixed` に切り替えて移動し、`state.maskToolbarPos` に保存する。`openAssetDetail` / `closeAssetDetail` / `toggleMaskEditor` で位置と最小化状態をリセットする。
- マスク編集の ON/OFF は、画像詳細ビューア上部の画像外コントロールバーにある常設ボタンで切り替える。マスクキャンバス上に切り替え文字を重ねない。
- 最小化中のツールバー見出しはドラッグ対象にせず、展開ボタン以外はポインターを透過させてマスク描画を妨げない。
- マスク編集中は画像上部に「マスク編集モード」と現在のツール・ブラシサイズを表示する。
- 画像詳細ビューアの下部情報と「選択切替」「この画像からブランチング」は画像外の通常フローに置き、縦長画像や画像読み込み失敗時でもマスクキャンバスに重ねない。
- マスク編集ツールバー内の入力欄にフォーカスがある間は、画像詳細ビューのショートカットを発火させない。Positive prompt では Space や Ctrl+A などの標準入力操作を優先する。
- マスク編集ツールバーの Positive prompt 直下にあるバッチサイズスライダーは、通常の生成フォームの `batchSize` draft と同期し、マスク付き img2img 送信時にも同じ値を使う。

## 逐次生成・停止

- 通常生成の `batchSize` は ComfyUI 内部の大きな1バッチではなく、GURUGURU 側で `batch_size=1` の `generation_jobs` に分解してキュー投入します。
- 各ジョブは同じ ComfyUI `client_id` を使い、サーバー側 WebSocket で `executed` / `execution_success` / `execution_interrupted` / `execution_error` を監視します。WebSocket通知を取り逃がした場合も、UIの自動 `collect` polling で回収します。
- 分割ジョブの seed は round に保存した先頭 seed を基準に、`batchIndex` ごとに `seed + batchIndex` を使います。asset の seed は各ジョブの実 seed です。
- 停止ボタンは未実行・待機中 prompt を ComfyUI queue から削除し、実行中 prompt がこの round のものと判断できる場合は ComfyUI `/interrupt` を呼びます。保存済み asset はそのまま残り、停止後も選択や画像からのブランチングに使えます。
- 複数アプリや別ユーザーが同じ ComfyUI を共有する運用では、`/interrupt` が現在実行中の ComfyUI ワークフローへ作用する点に注意します。

## イテレーションツリーのスクロール保持

- イテレーションツリー (`.iteration-tracker`) は `render()` で `app.innerHTML` ごと再生成されるため、DOM 再生成前後で `scrollLeft` / `scrollTop` を保存・復元する必要があります。
- `render()` はデフォルトでスクロール位置を保存・復元します。Round 選択 (`selectRound`) や画像詳細オープン (`openAssetDetail`) でも `render()` をそのまま使い、スクロール保持を有効にします。
- `render({ preserveIterationScroll: false })` はスクロールを意図的にリセットしたい場合だけ使います。
- プロジェクト切替 (`openProject`) や Home 戻り (`loadHome`) では、別プロジェクトの古いスクロール位置を引き継がないよう `state.iterationScroll = null` を明示的にリセットします。
- `scrollIntoView()` で active ノードに寄せる修正は、ユーザーの手動スクロール位置を維持する要件と逆になるため入れません。
