# 操作メモ

GURUGURU の操作方法や注意点を追記するためのメモです。大きな仕様書ではなく、実際に開発・運用するときに必要な手順と落とし穴を残します。機能の内部実装・設計は `Docs/Reference-*.md`(随時更新するリファレンス)や `Docs/Done/`(完了機能の設計ログ)を参照してください。

## 開発時の基本操作

- 依存関係は `bun install` で用意します。Bun の想定バージョンは `package.json` の `packageManager` に合わせます。`.guruzip`のネイティブ展開にRust 1.88以上も必要です。
- ビルドは `bun run build` を使います。Rustだけを先に確認する場合は`bun run build:native`を使います。
- ホットプレビュー付きの開発起動は `bun run dev` を使います。`src/` と `scripts/` を監視し、変更時に再ビルド・サーバー再起動・ブラウザ自動リロードを行います。
- 一回だけビルドして起動する場合は `bun run dev:once` を使います。
- ビルドとサーバー起動 smoke を含む確認は `bun run check` を使います。
- 型チェックは `bun run typecheck`(`tsc --noEmit`、strict)を使います。コード変更後はゼロエラーを維持します。
- テストは`bun run test`を使います（native build済みなら`bun test`だけでも可）。
- CSS のブラウザ基準は **Chrome 150** を前提とします。`:has()`・`mask` 合成・CSS anchor positioning などの新しい CSS 機能をフォールバックなしで使ってよく、後方互換のための prefix / 代替実装は不要です(`src/client/styles.css` 冒頭コメントにも明記)。

## データ保存場所

- 通常実行時のユーザーデータは OS のユーザーデータディレクトリに保存されます。Windows では `%LOCALAPPDATA%\GURUGURU` が既定です。
- `GURUGURU_DATA_DIR` を指定する場合は、必ずこのリポジトリ外のパスにします。
- `app.db`、生成画像、プロンプト、ワークフロースナップショット、サムネイルなどのランタイムデータはリポジトリに置きません。
- リポジトリ直下や `data/` に実運用データベースを作らないでください。

## テスト・検証時の注意

- データベースを初期化するコマンドやテストでは `GURUGURU_TEST_DB=1` を設定します。
- テスト用データの保存先を明示したい場合は `GURUGURU_TEST_DATA_DIR` を使います。
- 本番のユーザーデータディレクトリや実運用の `app.db` を、テスト・調査・要約目的で読み込まないでください。
- コマンドがリポジトリ内のデータディレクトリを拒否した場合は、外部のテスト用ディレクトリを指定してやり直します。
- 既定ポート `5177` はユーザーが `bun run start` などで起動する通常インスタンス用に空けておきます。エージェントが検証用にアプリを起動する場合は `PORT` に 5177 以外を指定し、`GURUGURU_TEST_DB=1` とリポジトリ外の `GURUGURU_TEST_DATA_DIR` を使います。
- 複数インスタンスを同時に動かす場合は、ポートだけでなくデータディレクトリも分けます。同じ本番データディレクトリを複数プロセスから使う運用は避けます。
- UI レイアウト検証では、サイドバーやモーダルの崩れを確認するためにブラウザ viewport を一時的にワイド表示(例: 1680x920)へ切り替えてよいです。確認後は viewport 設定を戻します。
- `curl` でサーバー疎通確認するときは必ず `--max-time 5` を付ける。dev server・watch・test・Bun script などハングしうるコマンドは長時間待たず、1つの確認が 5〜10秒以上返らない場合は中断して原因を報告する。サーバー疎通確認では全 URL が必ずレスポンスを返すことを確認する(ディレクトリ URL でハングしないこと)。
- 本番 ComfyUI(既定ポート 8188)も本番ユーザーデータとして扱う。生成内容が見えるエンドポイント — `/history`(ワークフロー・プロンプト・シード)、`/view`(画像)— は読まず、input/output ディレクトリも閲覧しない。疎通確認レベル(`/queue`・`/system_stats`・`/object_info`)は可。エラー診断は、UI に表示されるエラーメッセージの共有をユーザーに依頼するか、下記のテスト用 ComfyUI インスタンスで再現して行う。本番インスタンスへの状態変更(`/free`・`/interrupt` 等)は事前にユーザーへ確認する。
- スマート選択(WebSAM)とポーズ検出のモデルは、既定でローカルAPIプロキシから本リポジトリの公開GitHub Release Assetを取得します。GitHubトークンの設定は不要です。

## テスト用 ComfyUI コンテナ

生成パイプラインの実機検証は、本番 ComfyUI(8188)ではなく `sandbox/` の GPU コンテナ(port 8288)を使います。モデルだけを既存 models ルートから read-only 共有し、input/output/temp/user は専用 named volume です。

```powershell
Copy-Item sandbox/.env.example sandbox/.env
# 初回だけ .env の GURUGURU_MODELS_DIR を実環境に合わせる
docker compose --env-file sandbox/.env -f sandbox/compose.yaml up -d --build
bun sandbox/scripts/check-comfy.mjs http://127.0.0.1:8288
bun run start:test
```

- GURUGURU の設定画面で ComfyUI URL を `http://127.0.0.1:8288` にする。RunPod 利用時は同じ欄を private URL に替える。
- `bun run start:test` は port 5199、test DB、リポジトリ外の専用 data dir を強制する。
- GPU 負荷を抑えるため、検証は低負荷パラメータ(例: 512x288 / 6 steps / batch 1)で行う。
- 停止は `docker compose --env-file sandbox/.env -f sandbox/compose.yaml down`。テスト生成物も消す場合だけ `--volumes` を追加する。
- Dockerfile/Compose の共通 mount 契約、RunPod 切替、Fountain→PPTX 手順は [`sandbox/README.md`](sandbox/README.md) を参照する。
- Anima 検証では Base v1.0、`qwen_3_06b_base`、`qwen_image_vae` を外部テスト models ルートへ置く。既定は `er_sde` / `simple` / 30 steps / CFG 4。Anima 用 LoRA だけを使用し、Chroma/SDXL 用 LoRA、Chroma ControlNet、PuLID-Flux は流用しない。

## マスク・ポーズ添付、画像貼り付け

- マスク編集・inpaint・ポーズ ControlNet 添付の内部実装(合成ルール・`patchInpaintLatentPath`・roleMap 接続トレース等)は [`Docs/Reference-MaskAndPoseAttachments.md`](Docs/Reference-MaskAndPoseAttachments.md) を参照してください。

- 落とし穴: 合成 PNG が 16MB を超えると生成前にクライアント側で中断されます(トースト表示)。`paste_sources/` のファイルは手動で削除しないでください(過去ラウンドの `request_json` からも参照されます。不要になったファイルはサーバ起動時に自動掃除されます、`purgeOrphanPasteSources`)。外部サイトからの画像 URL ドラッグ(ファイル実体なし)は taint 回避のため拒否されます。

## Chronicle バー(脚本タイムライン・吹き出し一括配置)

- Chronicle バーは、**Book かつ Fountain 脚本(MangaScript)が1件以上取り込まれている**プロジェクトでのみ、ページ編集 lightbox 下部に表示されます。脚本の無い Book やコマ割りページでは表示されません。

## Fountain 自動漫画(MangaPlanV2 / ネーム監督)

- `/api/projects/:id/script-manga-runs` に `planningMode: "llm"` を渡すと、設定済みOpenAI互換LLMがコマ生成前にネームを演出する。人物一貫性には `characterBible` へ髪・体格・服・固有装備を固定記述し、品質優先時は `longEdge` / `steps` / `cfg` / `sampler` / `scheduler` を明示する。
- 先頭ページだけの比較は `pageLimit`、Chroma用スタイルLoRAは `loras: [{ name, strength }]` を指定する。SDXL/Illustrious用LoRAはChromaへ流用しない。
- 手動でオブジェクト(自動生成された吹き出し等)を移動/リサイズ/回転/尻尾変更すると、その placement は自動的に `auto_layout_locked` になります。ロック済みは「再配置」の対象から外れ、Chronicle バー上でも点線枠で示されます。ロックは Beat プレビュー内の個別解除、またはツールバーの「ロック解除」一括ボタンで戻せます。
- 「再配置」は現在ページの**ロックされていない**吹き出しだけを新しい seed で組み直します。手動で動かした(ロックされた)吹き出しは動きません。
- Fountain の再取り込み(`POST /api/scripts/:id/revisions`)後、削除された行は Chronicle 上で `orphaned` 状態(赤枠・半透明)として残ります。**自動削除はされません**。不要な吹き出しは手動で削除してください。一致する行(同一 `source_hash`)は割り当て・吹き出しをそのまま維持し、新規行は未配置として追加されます。
- 配置ソルバー(候補生成・スコア・フォールバック規則)の内部仕様は [`Docs/Reference-DialogueAutoLayout.md`](Docs/Reference-DialogueAutoLayout.md) を参照してください。機能の全体像・実装経緯は [`Docs/Done/Feature-ChroniclePageFlow.md`](Docs/Done/Feature-ChroniclePageFlow.md) を参照してください。
- Fountain一括漫画生成は`POST /api/projects/:projectId/script-manga-runs`。既定は4コマ/ページ、batch 1、各コマ長辺1024px。作成時のscript revisionとMangaPlanV2が固定され、後からFountainを再取り込みしても既存runの根拠は変わりません。
- `planningMode:"provided"`で1コマに複数発話(`dialogueOrderIndexes`が2件以上)を割り当てると、レタリング時に先着の発話がスペースを占有し後続が最小可読サイズ(0.016)を割って**run作成全体が422で失敗する**ことがある(エラーは違反1件のみ報告、ページ/コマの特定情報なし)。短い発話でも小さいレイアウトスロットでは縦書きの列高さが響いて割れる。事前に`POST /api/text-layout`で実フォント寸法を検証し、違反コマを1発話1コマへ分割/独立ページへ昇格→再検証を収束するまで反復するとよい。分割時は元の複数発話ぶんの引用文をそのまま複製しない(他キャラクター名が`characterIdsForText`経由でcastへ紛れ込む)。詳細と検証手順は[`Docs/Reference-DialogueAutoLayout.md`](Docs/Reference-DialogueAutoLayout.md)§12。
- Script画面の「MangaPlan V2 / 一括生成」は必ずprepareから始まり、warning確認→承認→生成開始→更新/再開/キャンセル→候補確認・コマ再生成・採用→PNG/PPTX/ORA書き出しを操作できます。API利用時は`generateImages:false`でprepareし、planを`GET/PATCH /api/script-manga-plans/:planId`で確認してからrunの`approve`→`start`を呼びます。run所有ページのlayout変更・直接削除はできません。
- VLM監査を使う場合は、`HauhauCS/Gemma-4-E2B-Uncensored-HauhauCS-Aggressive`のQ6_K_P GGUFと対応mmprojをLM Studioへ事前importします。model fileはrepositoryやGURUGURU data dirへ置きません。設定は`GET/PUT /api/settings/vlm-audit`、readiness確認は`GET /api/vlm-audit/status`です。既定のidentity参照は3件、設定範囲は0〜6件です。
- `auditMode:"vlm"`は全画像生成完了後、ComfyUI global queueがidleなら`/free`でComfy modelを解放し、LM Studio VLMをon-demand loadして全候補を直列監査し、終了後にunloadします。両modelを同時常駐させません。共有ComfyUIのmodelを解放したくない運用では`auditMode:"manual"`を選んでください。queueがbusyなら監査はdeferredになり、`/free`は呼ばれません。
- VLMはreasoning offで人物同一性・行動・偽文字・連続性を採点しますが、利用不可・不正応答時は人間reviewへfail-openします。manual/VLMとも候補は自動採用されず、`POST /api/script-manga-tasks/:taskId/select`の`assetId`で人が採用します。completed runは`POST /api/script-manga-runs/:runId/export`で所有ページだけをPNG/JPEG/PPTX/ORAへ書き出せます。
- 再起動後はrunの`resume`、中止は`cancel`、失敗taskのコマ全体再生成は`POST /api/script-manga-tasks/:taskId/retry`です。進捗同期は`GET /api/script-manga-runs/:runId`。詳細は[`Docs/Feature-MangaPlanV2.md`](Docs/Feature-MangaPlanV2.md)と[`Docs/Reference-GenerationPipeline.md`](Docs/Reference-GenerationPipeline.md)を参照してください。

## キャラクター Reference Set

- ページ一覧上部の「レファレンスコーナー」は集計サマリです。「開く」から詳細モーダルへ進み、キャラクタータブで対象を切り替えてvariant / model familyごとのfaceとfull_bodyを管理します。自動生成・再生成・アップロードはいずれも候補作成であり、自動採用されません。候補を比較し、人が承認してください。
- Chroma Readyにはface、Anima Readyにはface + full_bodyの承認が必要です。外見設定を変更して新versionを作ると旧版は「要再生成」になりますが、既存Manga Runは承認時のset/version/checksumへ固定されます。
- Anima In-Context adapter / nodeは非商用条件の任意導入で、GURUGURUへ同梱しません。interactive生成の未導入fallbackは警告を確認し、自動漫画はpreflightを解消してから承認します。詳細は[`Docs/Reference-AnimaPreset.md`](Docs/Reference-AnimaPreset.md)。
- 実機比較は本番8188を使わず、隔離8288へ`sandbox/scripts/check-reference-set.mjs`を実行します。出力先はrepository外を指定し、長辺768の採点後に勝者だけ1024へ昇格します。製品既定は長辺768 / batch 1です。顔寄りはface、全身・遠景はfull_bodyの一枚を優先し、1024のwaist-upは小物重複を確認してから採用してください。
