# 操作メモ

GURUGURU の操作方法や注意点を追記するためのメモです。大きな仕様書ではなく、実際に開発・運用するときに必要な手順と落とし穴を残します。機能の内部実装・設計は `Docs/Reference-*.md`(随時更新するリファレンス)や `Docs/Done/`(完了機能の設計ログ)を参照してください。

## 開発時の基本操作

- 依存関係は `npm install` で用意します。
- ビルドは `npm run build` を使います。
- 開発起動は `npm run dev` を使います。これはビルド後に `dist/server/index.js` を起動します。
- 構文チェックを含む確認は `npm run check` を使います。
- 型チェックは `npm run typecheck`(`tsc --noEmit`、strict)を使います。コード変更後はゼロエラーを維持します。
- 実行には Node.js 24 以上が必要です。
- CSS のブラウザ基準は **Chrome 150** を前提とします。`:has()`・`mask` 合成・CSS anchor positioning などの新しい CSS 機能をフォールバックなしで使ってよく、後方互換のための prefix / 代替実装は不要です(`src/client/styles.css` 冒頭コメントにも明記)。

## データ保存場所

- 通常実行時のユーザーデータは OS のユーザーデータディレクトリに保存されます。Windows では `%LOCALAPPDATA%\GURUGURU` が既定です。
- `GURUGURU_DATA_DIR` を指定する場合は、必ずこのリポジトリ外のパスにします。
- `app.db`、生成画像、プロンプト、ワークフロースナップショット、サムネイルなどのランタイムデータはリポジトリに置きません。
- リポジトリ直下や `data/` に実運用データベースを作らないでください。

## テスト・検証時の注意

- データベースを初期化するコマンドやテストでは `GURUGURU_TEST_DB=1` を設定します。
- テスト用データの保存先を明示したい場合は `GURUGURU_TEST_DATA_DIR` を使います。
- 本番のユーザーデータディレクトリや実運用の `app.db` を、テスト・調査・要約目的で読み込まないでください。
- コマンドがリポジトリ内のデータディレクトリを拒否した場合は、外部のテスト用ディレクトリを指定してやり直します。
- 既定ポート `5177` はユーザーが `npm start` などで起動する通常インスタンス用に空けておきます。エージェントが検証用にアプリを起動する場合は `PORT` に 5177 以外を指定し、`GURUGURU_TEST_DB=1` とリポジトリ外の `GURUGURU_TEST_DATA_DIR` を使います。
- 複数インスタンスを同時に動かす場合は、ポートだけでなくデータディレクトリも分けます。同じ本番データディレクトリを複数プロセスから使う運用は避けます。
- UI レイアウト検証では、サイドバーやモーダルの崩れを確認するためにブラウザ viewport を一時的にワイド表示(例: 1680x920)へ切り替えてよいです。確認後は viewport 設定を戻します。
- `curl` でサーバー疎通確認するときは必ず `--max-time 5` を付ける。dev server・watch・test・npm script などハングしうるコマンドは長時間待たず、1つの確認が 5〜10秒以上返らない場合は中断して原因を報告する。サーバー疎通確認では全 URL が必ずレスポンスを返すことを確認する(ディレクトリ URL でハングしないこと)。
- 本番 ComfyUI(既定ポート 8188)も本番ユーザーデータとして扱う。生成内容が見えるエンドポイント — `/history`(ワークフロー・プロンプト・シード)、`/view`(画像)— は読まず、input/output ディレクトリも閲覧しない。疎通確認レベル(`/queue`・`/system_stats`・`/object_info`)は可。エラー診断は、UI に表示されるエラーメッセージの共有をユーザーに依頼するか、下記のテスト用 ComfyUI インスタンスで再現して行う。本番インスタンスへの状態変更(`/free`・`/interrupt` 等)は事前にユーザーへ確認する。
- スマート選択(WebSAM)は既定で `/api/websam-models` から GitHub Release を取得します。本リポジトリは private のため、サーバー起動時に `GURUGURU_GITHUB_TOKEN` / `GH_TOKEN` / `GITHUB_TOKEN` のいずれかを設定してください(token はサーバー内だけで使い、ブラウザへは渡しません)。

## テスト用 ComfyUI インスタンス(DL 済み Desktop 版の流用)

生成パイプライン(workflow patch・ControlNet・統合 switch 等)の実機検証は、本番 ComfyUI(8188)ではなく、ダウンロード済み Desktop 版 ComfyUI のコードと Python 環境を流用した分離インスタンス(port 8288)で行います。

- コード: `$env:LOCALAPPDATA\Comfy-Desktop\ComfyUI-Installs\ComfyUI\ComfyUI`
- Python: 同ディレクトリの `standalone-env\python.exe`
- モデルは `%APPDATA%\Comfy Desktop\shared_model_paths.yaml` 経由で Desktop 版と共有(ダウンロード不要)

```powershell
# 1. リポジトリ外にテスト用 base ディレクトリを用意(custom_nodes が無いと起動時 FileNotFoundError)
$base = "$env:TEMP\comfy-test-8288"
New-Item -ItemType Directory -Force "$base\custom_nodes", "$base\input", "$base\output", "$base\temp"
# 統合 switch 検証では未使用枝の LoadImage にも実在ファイルが必要 → input にダミー画像を置く

# 2. 起動
$comfy = "$env:LOCALAPPDATA\Comfy-Desktop\ComfyUI-Installs\ComfyUI"
& "$comfy\standalone-env\python.exe" "$comfy\ComfyUI\main.py" `
  --port 8288 --base-directory $base `
  --extra-model-paths-config "$env:APPDATA\Comfy Desktop\shared_model_paths.yaml"

# 3. 疎通確認
curl --max-time 5 http://127.0.0.1:8288/system_stats
```

- GURUGURU 側はテストインスタンス(`PORT` ≠ 5177・`GURUGURU_TEST_DB=1`・外部 `GURUGURU_TEST_DATA_DIR`)を起動し、設定画面の ComfyUI URL に `http://127.0.0.1:8288` を設定して接続します。
- GPU 負荷を抑えるため、検証は低負荷パラメータ(例: 512x288 / 6 steps)で行います。
- テストインスタンスの output/input はテスト専用なので閲覧・削除して構いません。
- Desktop 版更新後に `standalone-env` から torch 等が消えることがあります。`requirements-nvidia.txt`(ピン留め torch)と `ComfyUI/requirements.txt` を `standalone-env\python.exe -m pip install -r` で復元すれば起動できます。

## マスク・ポーズ添付、画像貼り付け

- マスク編集・inpaint・ポーズ ControlNet 添付の内部実装(合成ルール・`patchInpaintLatentPath`・roleMap 接続トレース等)は [`Docs/Reference-MaskAndPoseAttachments.md`](Docs/Reference-MaskAndPoseAttachments.md) を参照してください。
- 画像貼り付け(Paste & Transform)は、D&D / Ctrl+V / サムネイルドラッグでアセットへ画像を「添付」でき、生成時にクライアントが見たまま合成して送信します(親画像・ツリー構造は不変)。設計・実装の詳細は [`Docs/Done/Feature-ImagePaste.md`](Docs/Done/Feature-ImagePaste.md) を参照してください。
- 落とし穴: 合成 PNG が 16MB を超えると生成前にクライアント側で中断されます(トースト表示)。`paste_sources/` のファイルは手動で削除しないでください(過去ラウンドの `request_json` からも参照されます。不要になったファイルはサーバ起動時に自動掃除されます、`purgeOrphanPasteSources`)。外部サイトからの画像 URL ドラッグ(ファイル実体なし)は taint 回避のため拒否されます。
